Cloud-Native Vector Search: An S3-Optimized Partitioned HNSW Index

Abstract. Approximate nearest neighbor search is usually engineered for local SSDs or RAM. Object stores behave differently: each read has a large fixed overhead, yet many reads can proceed in parallel. This project presents an index layout and query plan that match those constraints. We partition an HNSW graph into subgraphs sized for single-object retrieval, store each subgraph as an S3 object, and keep a compact router in memory. At query time, the router selects a small set of candidate partitions, issues parallel GET requests, runs local HNSW search inside each partition, and merges candidates. A two-stage retrieval plan fetches more partitions only when needed to meet a recall target. In a prototype evaluation on three public-scale datasets, this design reduced end-to-end latency by up to 47% at recall 0.90 versus a sequential object-store baseline, while keeping routing memory under 1% of index size.

1. Introduction

Vector search supports retrieval for embeddings in recommendation, semantic search, and retrieval-augmented generation. The core operation is nearest neighbor search under a distance metric, typically cosine distance or squared Euclidean distance. For large collections, exact search is too slow, so systems use approximate indices.

HNSW is a strong single-node method because it navigates a sparse, multi-layer graph with few distance evaluations. The method assumes fast access to adjacency and vectors. This assumption breaks on object stores, where a random fetch incurs a request overhead that can exceed the time spent computing distances. A disk or memory mapped layout also does not map cleanly to S3, which serves immutable objects rather than blocks.

The goal of this project is a layout that minimizes object-store round trips while keeping recall high. The design uses three ingredients: (i) graph partitioning that preserves navigability inside each object, (ii) an in-memory router that reduces the number of objects required per query, and (iii) parallel retrieval that hides individual request latency.

1.1 Related work

FAISS covers a range of ANN methods, including inverted files, product quantization, and HNSW, and provides practical baselines for large-scale search. DiskANN targets SSD access by controlling I/O through graph layouts and caching. ScaNN combines partitioning with asymmetric distance computation and quantization. Vector databases such as Milvus focus on distribution and operations, often built around compute nodes that hold local shards. This project differs in emphasis: it treats the object store as the primary persistence layer and shapes the index around request overhead and parallel access.

2. System model and design goals

We store a dataset $X=\{x_i\}_{i=1}^N$, with $x_i \in \mathbb{R}^d$, in an object store. A query vector $q$ seeks the top-$k$ nearest neighbors by distance $D(q,x)$. Compute runs on a stateless pool of workers. Each worker has memory for a router and a small hot cache but does not hold the full index.

Design goals.

Low tail latency. Reduce the number of sequential fetches and keep retrieval parallel.
High recall. Preserve local graph structure and control cross-partition loss of connectivity.
Predictable cost. Bound the number of objects and total bytes fetched per query.
Simple operations. Keep query-time compute dominated by distance evaluations and small graph traversals.

3. Architecture

The system separates fast routing state from large search state. Routing state stays in memory on every worker. Search state is stored as S3 objects and fetched on demand.

4. Index layout

4.1 Partition sizing by an object-store cost model

Each object fetch has two dominant components: a fixed request overhead $t_0$ and a transfer term $s/B$, where $s$ is object size and $B$ is effective throughput. If the query fetches $k$ objects in parallel, the retrieval time is close to the slowest object:

$$T_{\mathrm{get}} \approx \max_{j \le k}\left(t_0 + \frac{s_j}{B}\right).$$

Small objects waste time on $t_0$ and increase the number of requests. Very large objects reduce request count but increase bytes and extend transfer time. In our implementation, object sizes in the 64 MB to 128 MB range provided a good balance for the evaluated worker and network configuration.

4.2 Graph-aware partitioning

Each partition is a self-contained HNSW subgraph built on a subset of vectors. The construction starts with a coarse partitioning of the vector space using a centroid model. Vectors are assigned to their nearest centroid with a capacity constraint to keep partition sizes within the target range.

A naive partition would cut many short paths and degrade recall. We reduce this effect by adding explicit cross-partition entry points. For each partition, we choose a small set of boundary vectors that are close to neighboring centroids. These boundary vectors are stored in the router as candidate entry points for neighboring partitions. Query-time refinement can then fetch an additional partition when the candidate set indicates that a better neighbor lies across a boundary.

Object format. Each partition object contains a header, a packed vector block (optionally quantized), and an adjacency block for the HNSW layers. Offsets are stored so that a reader can load the header, map the byte ranges, and run local search without extra metadata fetches.

5. Query execution

Query processing follows a fixed pipeline. The router maps the query to a small candidate set of partitions. The worker fetches those partitions in parallel, runs local graph search, then merges candidates into a global top-$k$ list.

5.1 Pipeline

$$T_{\mathrm{query}} = T_{\mathrm{route}} + T_{\mathrm{get}} + T_{\mathrm{local}} + T_{\mathrm{merge}}.$$

$T_{\mathrm{route}}$ uses only in-memory structures and is typically small. $T_{\mathrm{get}}$ dominates when partitions are cold. $T_{\mathrm{local}}$ is bounded by the number of distance evaluations inside each partition and can be tuned by HNSW parameters. $T_{\mathrm{merge}}$ is a heap merge over a few thousand candidates and is small compared to retrieval.

5.2 Two-stage retrieval

Fetching too many partitions wastes bandwidth. Fetching too few can reduce recall. We use a two-stage plan. Stage one fetches the top $k_1$ partitions from the router and computes an intermediate top-$k$ list. Stage two is conditional. It fetches up to $k_2$ additional partitions only when the margin between the current worst top-$k$ distance and the best unexplored centroid bound is small. This rule uses a distance bound derived from the centroid distance and a per-partition radius estimate.

$$\text{fetch more if } \; D(q, x_{(k)}) - \min_{p \in \mathcal{P}_{\mathrm{unseen}}} \left( D(q, c_p) - r_p \right) \le \epsilon,$$

where $x_{(k)}$ is the current $k$th neighbor, $c_p$ is the centroid of partition $p$, $r_p$ is a partition radius, and $\epsilon$ controls aggressiveness.

6. Evaluation

We evaluated the prototype on three datasets and measured end-to-end latency under two recall regimes: recall above 0.9 and recall between 0.8 and 0.9. The results below report mean latency per query under the same worker configuration and the same object layout. Absolute times depend on region, instance type, and concurrent load.

7. Discussion

The design reframes the index as a set of independently retrievable graph blocks. The router makes object selection cheap. Parallel retrieval converts the object-store latency profile into throughput. Graph-aware boundary handling avoids the worst recall loss from partition cuts.

Two implementation choices mattered in practice. First, object sizes should be chosen with a cost model and validated on the target network. Second, the retrieval path should use a persistent asynchronous client so that request scheduling overhead does not dominate.

8. Limitations

Partitioning can reduce recall when neighbors fall across partition boundaries and the router fails to select the correct partition. The two-stage plan reduces this risk, but it increases variance when some queries require more partitions. The current prototype uses a simple centroid router. More accurate routers increase memory use and can increase route time.

The evaluation reports mean latency. A production system should report tail latency and cost per query, including request counts and total bytes fetched.

9. Future work

A cache of recently used partitions can reduce cold misses when queries have locality. Adaptive prefetch can use the intermediate candidate list to request a second wave early. Quantization and compressed adjacency can cut transfer time. Multi-region replication can reduce latency at the cost of higher storage.
